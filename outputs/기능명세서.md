## 1. 인증

### 1.1 GitHub OAuth 로그인

**기능 설명**  
사용자가 GitHub OAuth 2.0을 통해 인증하고 액세스 토큰을 발급받아 앱에 로그인한다.

**정상 흐름**

1. 사용자가 `/login` 진입 → "GitHub으로 로그인" 버튼 클릭
2. 서버가 `state` 파라미터(CSRF 방지용 난수) 생성 → 세션에 저장
3. GitHub OAuth URL로 리다이렉트 (`scope=repo,user:email`)
4. GitHub 인증 완료 → `/auth/callback?code=xxx&state=yyy` 수신
5. 서버가 `state` 검증 → code로 access_token 교환 (서버 사이드)
6. access_token을 `httpOnly` Secure 쿠키에 저장
7. 사용자 정보(`/user`) 조회 → DB 또는 세션에 저장
8. 레포 선택 여부 확인 → `/select-repo` 또는 `/workspace`로 이동

**예외/실패 케이스**

| ID   | 케이스         | 원인                            | 처리                                      |
| ---- | -------------- | ------------------------------- | ----------------------------------------- |
| A-01 | state 불일치   | CSRF 공격 또는 세션 만료        | 에러 페이지 표시, 재시도 유도             |
| A-02 | code 교환 실패 | 만료된 코드(10분 내 미사용)     | "인증이 만료되었습니다" 메시지 + 재로그인 |
| A-03 | GitHub API 503 | GitHub 서비스 장애              | GitHub 상태 링크 제공, 재시도 버튼        |
| A-04 | scope 거부     | 사용자가 `repo` scope 거부      | 필요 권한 설명 후 재요청                  |
| A-05 | 쿠키 설정 실패 | Safari ITP / 서드파티 쿠키 차단 | `SameSite=Lax` + 동일 도메인 설정         |
| A-06 | 토큰 만료      | GitHub token 무효화             | 자동 감지 → 재로그인 흐름                 |

**보안 요구사항**

- `state` 파라미터: 최소 32바이트 CSPRNG 값
- access_token: 클라이언트 JS에서 접근 불가 (`httpOnly` 쿠키)
- 쿠키 속성: `Secure; HttpOnly; SameSite=Lax; Path=/`
- 최소 권한: `repo` scope만 요청 (관리자 권한 불필요)
- 토큰 갱신: GitHub OAuth는 refresh_token 없음 → 만료 감지 시 재인증

## 2. 레포 연동

### 2.1 레포 선택

**기능 설명**  
사용자의 GitHub 레포 목록을 조회하여 노트 저장소로 사용할 레포를 선택한다.

**정상 흐름**

1. `/user/repos?per_page=100&sort=pushed` API 호출
2. 레포 목록 표시 (이름, 공개/비공개, 마지막 수정일)
3. 사용자 선택 → `localStorage`에 `{owner, repo}` 저장
4. 레포 루트에 `.notegraph/config.json` 존재 여부 확인
5. 없으면 초기화 파일 자동 생성 (첫 사용 설정)
6. `/workspace`로 이동

**예외/실패 케이스**

| ID   | 케이스                   | 처리                                                   |
| ---- | ------------------------ | ------------------------------------------------------ |
| R-01 | 레포 0개                 | "새 레포 만들기" 링크 제공 (GitHub로 이동)             |
| R-02 | 레포 100개 초과          | 페이지네이션 또는 검색 기능 제공                       |
| R-03 | 선택 레포 접근 권한 없음 | 403 감지 → "접근 권한이 없는 레포입니다"               |
| R-04 | config.json 생성 실패    | 레포가 비어있지 않은 경우에도 폴백으로 메모리에서 시작 |

**개발 고려사항**

- 레포 목록 캐싱: 5분 TTL (stale-while-revalidate)
- 포크된 레포는 구분 표시 (선택 시 확인 다이얼로그)
- 아카이브된 레포는 읽기 전용 표시

---

### 2.2 파일 트리 로딩

**기능 설명**  
선택된 레포의 전체 파일 트리를 GitHub Git Trees API로 로딩한다.

**정상 흐름**

1. `GET /repos/{owner}/{repo}/git/trees/HEAD?recursive=1`
2. `.md` 파일 및 디렉토리만 필터링
3. 경로 기준으로 계층 구조 파싱
4. 사이드바 FileTree에 렌더링

**예외 케이스**

| ID   | 케이스            | 처리                                                 |
| ---- | ----------------- | ---------------------------------------------------- |
| R-05 | 파일 1,000개 초과 | GitHub API 최대 100,000 파일 지원; 초과 시 경고 표시 |
| R-06 | 빈 레포           | "첫 노트를 만들어보세요!" 빈 상태 UI                 |
| R-07 | 네트워크 오류     | 스켈레톤 UI + 재시도 버튼                            |
| R-08 | 레포 삭제됨       | 404 감지 → 레포 재선택 안내                          |

### 3.1 폴더 생성

**기능 설명**  
사이드바에서 새 폴더를 생성한다. (GitHub는 빈 폴더를 지원하지 않으므로 `.gitkeep` 파일로 대체)

**정상 흐름**

1. [+폴더] 버튼 또는 컨텍스트 메뉴 → 이름 입력 모달
2. 유효성 검사 통과 → GitHub API로 `{폴더명}/.gitkeep` 파일 생성
3. 파일 트리에 새 폴더 즉시 반영 (Optimistic UI)
4. 새 폴더 자동 선택 및 펼치기

**유효성 검사 규칙**

- 이름 필수, 최소 1자 최대 100자
- 허용 문자: 한글, 영문, 숫자, 하이픈(-), 언더스코어(\_), 공백
- 금지 문자: `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`, `.` 시작
- 동일 경로에 같은 이름 중복 금지 (대소문자 구분 없음)

**예외/실패 케이스**

| ID   | 케이스          | 처리                                   |
| ---- | --------------- | -------------------------------------- |
| S-01 | 중복 이름       | "이미 존재하는 이름입니다" 인라인 에러 |
| S-02 | 금지 문자 입력  | 실시간 필터링 또는 인라인 경고         |
| S-03 | GitHub API 실패 | Optimistic UI 롤백 + 에러 토스트       |
| S-04 | 이름 100자 초과 | 입력 차단 + "최대 100자" 안내          |

### 3.2 노트 생성

**정상 흐름**

1. [+노트] 또는 컨텍스트 메뉴 → 이름 입력 모달 (`.md` 자동 추가)
2. GitHub API로 frontmatter 포함 초기 파일 생성
3. 트리에 추가 → 해당 노트 자동 열기

created: { ISO8601 }

tags: []

### 3.3 이름 변경

**정상 흐름**

1. 컨텍스트 메뉴 → 이름 변경 → 모달
2. 기존 이름 선택된 상태로 표시
3. 확인 → GitHub API: 기존 파일 내용 GET → 새 경로로 PUT → 기존 파일 DELETE
4. v2: 해당 노트를 참조하는 `[[백링크]]` 자동 업데이트

**예외 케이스**

### 3.5 드래그앤드롭 이동

**라이브러리**: `@dnd-kit/sortable`

**정상 흐름**

1. 왼쪽 핸들(⠿) 클릭 후 드래그 시작 → 원본 아이템 반투명(opacity 40%)
2. **Case A — 아이템 사이 드롭**: 아이템 사이 공간에 2px 녹색 선 표시 → 드롭 시 같은 레벨 순서 변경
3. **Case B — 폴더 위 드롭**: 폴더 위 800ms 호버 → 폴더 자동 열림 + 녹색 하이라이트 → 드롭 시 하위로 이동
4. **Case C — 폴더 자체 드래그**: 폴더 이동 시 하위 파일 전체 경로 일괄 변경 (GitHub API 병렬 요청)
5. Optimistic UI (즉시 트리 반영) → API 실패 시 원상복구 + 에러 토스트

**예외 케이스**

| ID   | 케이스                         | 처리                      |

| S-13 | 동일 위치 드롭                 | 무동작                    |

| E-09 | 업로드 실패                       | 로컬 미리보기 유지 + 에러 표시 + 재시도 버튼     |
| E-10 | 오프라인 상태에서 이미지 붙여넣기 | 로컬 blob URL로 임시 저장, 온라인 복귀 시 업로드 |

## 5. 저장

### 5.1 자동 저장

**동작**

- 마지막 입력으로부터 2초 후 자동 저장 트리거
- 저장 상태 표시: `Unsaved changes` → `Saving...` → `Saved`
- GitHub API PUT으로 파일 업데이트 (SHA 필수 포함)

**예외/실패 케이스**

| ID    | 케이스                               | 처리                                                        |
| ----- | ------------------------------------ | ----------------------------------------------------------- |
| SV-01 | SHA 충돌 (다른 기기에서 동시 편집)   | 409 감지 → "외부에서 변경됨" 경고 → 머지 또는 덮어쓰기 선택 |
| SV-02 | 네트워크 오프라인                    | 자동 저장 정지, `Offline` 표시, 온라인 복귀 시 즉시 재시도  |
| SV-03 | GitHub API Rate Limit (5,000회/시간) | 요청 큐잉 + 재시도 로직 (exponential backoff)               |
| SV-04 | 429 Too Many Requests                | `Retry-After` 헤더 파싱, 대기 후 재시도                     |
| SV-05 | 저장 중 탭/브라우저 닫기             | `beforeunload` 이벤트로 경고 표시                           |
| SV-06 | 빈 노트 자동 저장                    | 제목과 본문 모두 비어있으면 저장 스킵                       |

### 5.2 수동 저장

**동작**

- `Cmd/Ctrl+S` 또는 저장 버튼
- 자동 저장과 동일한 API 흐름
- 이미 Saved 상태라도 버튼은 항상 응답 (확인 피드백 제공)

---

### 5.3 동시 편집 처리 (SHA 충돌)

**흐름**

```
기기A 편집 중 → 기기B에서 동일 파일 저장 → SHA 변경
기기A 저장 시도 → 409 Conflict 수신
→ 현재 내용 + 서버 내용 diff 표시
→ 사용자 선택: [내 버전으로 덮어쓰기] [서버 버전 사용] [수동 머지]
```

## 6. 수정 이력

### 6.1 개요

**설계 원칙**: 별도 백업 파일을 만들지 않는다. 모든 저장(자동/수동)이 GitHub commit으로 남으므로 Git 히스토리 자체가 이력이다.

**수정 이력 뷰어 진입**: 에디터 툴바 "이력 보기" 버튼 → 좌우 분할 뷰 (이력 목록 + diff 미리보기)

### 6.2 이력 목록

**표시 정보 (commit당)**

- 저장 시각 (상대적: "방금 전", "3시간 전", 날짜 초과 시 절대)
- 저장 유형: `자동저장` / `수동저장 (Cmd+S)` 배지
- 변경량: +N줄 / -N줄, 파일 크기

**정상 흐름**

1. GitHub API: `GET /repos/{owner}/{repo}/commits?path={filePath}` 호출
2. 최근 50개 커밋 목록 표시
3. 커밋 선택 → 해당 커밋과 이전 커밋 diff 조회
4. diff 미리보기 표시 (추가: 녹색, 삭제: 빨간색, 컨텍스트: 회색)

**예외/실패 케이스**

| ID   | 케이스                      | 처리                                        |
| ---- | --------------------------- | ------------------------------------------- |
| H-01 | 커밋 이력 없음 (첫 저장 전) | "아직 저장 이력이 없습니다" 빈 상태         |
| H-02 | 커밋 50개 초과              | 더 불러오기 버튼 (무한 스크롤)              |
| H-03 | diff 로딩 실패              | 해당 커밋 재시도 버튼                       |
| H-04 | 바이너리 파일 diff          | "바이너리 파일은 diff를 표시할 수 없습니다" |

### 6.3 버전 복원

**정상 흐름**

1. 이력 뷰에서 특정 커밋 선택 → "이 버전으로 복원" 버튼
2. 확인 다이얼로그: "현재 내용도 이력에 저장됩니다"
3. 해당 커밋의 파일 내용 GET → 현재 파일에 PUT (새 commit 생성)
4. 에디터에 복원된 내용 반영 + 성공 토스트

**예외 케이스**

| ID   | 케이스                | 처리                                                |
| ---- | --------------------- | --------------------------------------------------- |
| H-05 | 복원 중 네트워크 오류 | 롤백 없음 (복원 실패, 기존 내용 유지) + 에러 토스트 |
| H-06 | 복원 중 SHA 충돌      | 다른 기기에서 동시 수정 → 409 처리 흐름과 동일      |

### 6.4 커밋 메시지 규칙

모든 저장은 일관된 커밋 메시지 형식을 사용한다:

```
[autosave] 아이디어 정리.md — 2026-06-17 09:51
[save] 아이디어 정리.md — 2026-06-17 09:42
[restore] 아이디어 정리.md — restored to 2026-06-17 09:42
```

## 9. 보안 요구사항

### 9.1 인증 보안

| 항목           | 구현 방법                                  |
| -------------- | ------------------------------------------ |
| CSRF 방어      | OAuth state 파라미터 + SameSite 쿠키       |
| XSS 방어       | TipTap 내 HTML 콘텐츠 DOMPurify 새니타이징 |
| 토큰 보관      | httpOnly 쿠키 (JS 접근 불가)               |
| API 키 보호    | 서버 환경변수만 사용, .env.local 추가      |
| 레포 접근 검증 | 모든 GitHub API 요청 시 owner 검증         |

### 9.2 데이터 보안

| 항목                  | 구현 방법                                            |
| --------------------- | ---------------------------------------------------- |
| 전송 보안             | HTTPS 강제 (HSTS 헤더)                               |
| GitHub 파일 쓰기 검증 | SHA 파라미터 필수 포함 (임의 덮어쓰기 방지)          |
| 경로 탈출 방지        | `../` 포함 경로 차단, 화이트리스트 검증              |
| 이미지 업로드 검증    | MIME 타입 + 매직 바이트 검증 (확장자만으로 불충분)   |
| OpenAI 프록시         | 클라이언트가 OpenAI에 직접 요청 불가, 서버 경유 필수 |

### 9.3 입력 유효성 검사

- 모든 사용자 입력: 서버사이드 재검증 (클라이언트 검증 신뢰 금지)
- 파일명: 금지 문자 정규식 필터 (`/[<>:"/\\|?*\x00-\x1F]/`)
- 파일 크기: GitHub API 제한 100MB 이내, 앱 제한 10MB

## 부록: API Rate Limit 전략

| API             | 제한         | 전략                             |
| --------------- | ------------ | -------------------------------- |
| GitHub REST     | 5,000 req/hr | 파일 트리 캐싱, 불필요 요청 제거 |
| GitHub Contents | 5,000 req/hr | 자동저장 디바운스 2초            |
| OpenAI          | tier별 상이  | 요청 큐 + backoff                |

**예상 요청 수 (활성 사용 1시간 기준)**

- 파일 트리: 1회 (캐시 후 불변)
- 자동 저장: 최대 30회/시간 (2초 디바운스 기준)
- 노트 열기: 20회 (파일별 1회)
- AI 요약: 5회 (저장된 노트 수 기준)
- **합계: ~56회 / 5,000 제한** → 여유 충분
